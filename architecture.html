InferSched Architecture

A visual walkthrough of how InferSched accepts inference requests, distributes them across Routers and Engines, schedules continuous batches, and safely recovers from failures.

InferSched simulates model execution with a deterministic cost model. Its measurements describe scheduler behavior and distributed correctness, not real GPU inference throughput.

Who owns what?

FastAPI Gateway

Validates HTTP requests, assigns correlation IDs, applies bounded admission, and returns completed results.

Kafka

Retains request events and divides partitions among Routers through a consumer group.

Router Fleet

Owns coordination, durable attempts, Engine dispatch, retry, rebalance, and recovery.

Engine Fleet

Owns continuous batching, active sequences, paged KV memory, caches, and preemption.

PostgreSQL

Owns authoritative lifecycle state, ownership epochs, fenced finalization, and the transactional outbox.

Redis

Holds disposable coordination hints. Losing Redis cannot corrupt authoritative request state.

How one request moves through the system

  1. Accept: FastAPI validates the request, assigns request and trace IDs, and publishes protobuf to Kafka.
  2. Assign: Kafka gives the request partition to exactly one active Router in the consumer group.
  3. Fence: The Router records the request in PostgreSQL and obtains a partition epoch and attempt ID.
  4. Execute: The Router dispatches over gRPC. The Engine continuously batches the request and manages its KV pages.
  5. Finalize: PostgreSQL accepts the completion only when the partition, epoch, Engine, attempt, and state are current.
  6. Publish: The result and outbox row commit together. The relay publishes to Kafka after the database transaction.
  7. Return: The gateway correlates the Kafka result with the waiting HTTP request and returns it to the client.

Guided implementation breakdown

Follow these stages in order when learning, presenting, or extending the project. Each stage depends on the correctness boundary established by the stage before it. Expand a stage to see its objective, the implementation to inspect, and the evidence that proves it works.

0 Foundation and contracts Define the build, request schema, service boundaries, and test configurations. Implemented

Understand

The protobuf contract separates Router coordination from Engine execution. CMake provides default, ASan, and TSan configurations.

Read

proto/infersched.proto, CMakePresets.json, and cpp/CMakeLists.txt.

Evidence

The same request, fence, dispatch, progress, cancellation, registration, and heartbeat messages are used by integration tests and real processes.

1 Deterministic Engine core Build correct lifecycle, batching, cache, KV-allocation, and simulated-time primitives. Implemented

Understand

Requests move through explicit states. The deterministic runner advances a fake clock using a reproducible prefill and decode cost model.

Read

execution_state.cpp, deterministic_engine.cpp, cost_model.cpp, and paged_kv_cache.cpp.

Evidence

Transactional allocation, reference counts, state transitions, cache keys, impossible-request rejection, and zero-leak invariants have focused tests.

2 Continuous batching and KV pressure Admit new work between decode iterations and remain correct when memory fills. Implemented

Understand

The scheduler admits ready requests, reuses prefixes, grows decode pages, evicts cache entries, and preempts lower-value work when necessary.

Read

continuous_engine.cpp, request_queue.cpp, radix_prefix_cache.cpp, and result_cache.cpp.

Evidence

Seeded workloads verify mid-decode admission, true shared prefixes across different prompts, completion despite preemption, and zero KV leakage.

3 Concurrency and scheduler benchmarks Remove scale bottlenecks and measure scheduler overhead without mislabeling it as GPU throughput. Implemented

Understand

A heap-backed request queue and sharded registry replace avoidable global or quadratic paths. Candidate optimizations are kept only when measurements justify them.

Read

request_queue.hpp, sharded_registry.hpp, mpsc_ring.hpp, and scheduler_bench.cpp.

Evidence

The 100,000-request CPU path completes with zero leaks. TSan verifies concurrency, and the slower lock-free queue is documented as a rejected optimization.

4 Real process distribution Run Routers and Engines as separate processes with a stable network protocol. Implemented

Understand

Routers host registration and heartbeat RPCs. Engines host dispatch, cancel, and drain RPCs. Dispatch streams acknowledgement and progress events.

Read

router_service.cpp, engine_service.cpp, router_main.cpp, and engine_main.cpp.

Evidence

Process tests cover concurrent dispatch, stale epoch and attempt rejection, same-Engine overlap rejection, SIGKILL, replacement, and drain.

5 Durable at-least-once pipeline Make PostgreSQL authoritative and publish accepted results without an unsafe dual write. Implemented

Understand

Idempotent ingestion tolerates Kafka redelivery. Conditional finalization accepts only the current fence. A transactional outbox connects database state to Kafka publication.

Read

durable_store.cpp, request_ingestor.cpp, outbox_relay.cpp, offset_tracker.cpp, and durable_router_main.cpp.

Evidence

Live Kafka and PostgreSQL tests prove duplicate handling, poison-to-DLQ publication, one accepted result, restart recovery, broker acknowledgement, and safe contiguous offsets.

6 Multi-Router rebalance and recovery Scale coordination by Kafka partitions while fencing late work after ownership changes. Implemented

Understand

Kafka decides assignment, while PostgreSQL decides authority. New owners increment partition epochs, recover unresolved requests, and create new attempts.

Read

kafka_poller.cpp, durable_router_main.cpp, and docs/p6-multi-router-rebalancing.md.

Evidence

Chaos tests kill a Router and Engine, trigger consumer-group rebalance, flush Redis, and still require all authoritative results to complete exactly once in PostgreSQL.

7 Control plane, observability, and evidence Expose the system through HTTP and measure overload, latency, utilization, caches, and the full distributed path. Implemented

Understand

The gateway bounds pending work, publishes protobuf, consumes results, correlates traces, and reports metrics. Open-loop load arrives independently of completions.

Read

python/infersched_api/app.py, python/loadgen.py, histogram.cpp, and docs/benchmarks.md.

Evidence

Tests cover HTTP 429 and the live FastAPI path. Reproducible baselines report bucketed latency, utilization, cache hit rates, completion, and zero KV leaks.
